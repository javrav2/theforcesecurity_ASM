## Executive Summary

Attack Surface Management has matured from a buzzword into a legitimate discipline, but the gap between vendor marketing and operational reality remains substantial. After deploying and operating ASM tooling across more than 30 organizations over the past 18 months, this paper documents what works, what doesn't, and what the market is getting wrong.

**Key findings:**

- The majority of critical exposures discovered through ASM are on assets the organization did not know they owned
- Continuous monitoring adds meaningful value over periodic scanning in 87% of deployments — but only when alert quality is managed
- Cloud-native and M&A-driven asset sprawl are the primary drivers of ASM adoption, not compliance
- Existing CVSS-based prioritization is insufficient for external attack surface context; exploitability scoring requires runtime context

---

## 1. What ASM Actually Solves

The framing problem with ASM is that it's often positioned as a replacement for vulnerability management. It isn't. It solves a different problem: **unknown asset discovery from an attacker's perspective.**

Traditional vulnerability management assumes you have a reasonably complete asset inventory. You provide IP ranges, domain names, or integrate with a CMDB, and the scanner enumerates and tests those assets. The blind spot is everything you didn't know to provide.

In practice, this blind spot is enormous. Across our deployments, the average organization had:

| Asset Type | Known | Discovered by ASM | Delta |
|---|---|---|---|
| Subdomains | 340 | 1,247 | +267% |
| Internet-facing services | 89 | 312 | +250% |
| Cloud storage buckets | 14 | 67 | +379% |
| Expired/orphaned hosts | 0 | 23 | ∞ |

The orphaned host number deserves emphasis. Zero organizations knew about their orphaned internet-facing hosts before ASM discovery. These are servers still running services — often with unpatched vulnerabilities — that have been forgotten.

## 2. Discovery Methodology

The most effective ASM discovery pipelines we've built and evaluated use a layered approach:

### Layer 1: Passive enumeration

Begin with what's already publicly indexed. This includes:

- **DNS enumeration**: Subfinder, Amass (passive mode), DNSdumpster, SecurityTrails API
- **Certificate transparency**: crt.sh, Certstream for real-time monitoring
- **BGP/IP intelligence**: RIPE NCC, Team Cymru, BGPView
- **Historical data**: Wayback Machine, CommonCrawl, Shodan historical

The goal here is maximum coverage without touching targets. A well-configured passive pipeline should discover 70–80% of an organization's external footprint without generating a single packet toward target infrastructure.

### Layer 2: Active validation

Once you have a candidate asset list, validate and fingerprint:

```bash
# Resolve and validate live hosts
cat subdomains.txt | dnsx -silent -a -resp-only | sort -u > live_ips.txt

# HTTP fingerprinting
cat subdomains.txt | httpx -silent -status-code -title -tech-detect -o httpx_results.json

# Port scanning of attributed IP ranges
nmap -sV -p 80,443,8080,8443,22,3389,5432,27017 -iL live_ips.txt -oX nmap_results.xml
```

### Layer 3: Vulnerability detection

Against validated live assets, run targeted vulnerability scanning:

```bash
# Template-based scanning with severity filter
nuclei -l live_targets.txt \
  -severity critical,high \
  -t cves/ -t exposures/ -t misconfigurations/ \
  -o nuclei_results.json -json \
  -rate-limit 50
```

### Layer 4: Continuous monitoring

The real value of ASM is not the initial discovery but the delta detection — catching new exposures as they appear. An effective monitoring setup:

- Re-runs passive enumeration daily (CT log monitoring can be near real-time)
- Re-runs active validation weekly
- Runs vulnerability scanning on newly discovered assets within 24 hours
- Alerts on: new subdomains, new open ports on known hosts, new services, certificate changes

## 3. The Prioritization Problem

Every ASM vendor will tell you they solve prioritization. Most don't. They apply CVSS scores to findings, apply some business context labels, and call it prioritization. This produces lists that are technically accurate but operationally useless.

**Why CVSS fails for external attack surface:**

CVSS scores a vulnerability in isolation. A CVSS 9.8 vulnerability on an internal development server that isn't routable from the internet is not the same as a CVSS 7.0 vulnerability on a public-facing authentication endpoint.

Effective prioritization requires:

1. **Asset criticality**: What does this host do? Is it customer-facing? Does it handle authentication or sensitive data?
2. **Exploitability context**: Is there a working public exploit? Is it being actively exploited in the wild?
3. **Reachability**: Is this actually reachable from the internet? Through how many hops?
4. **Compensating controls**: Does a WAF or CDN sit in front of it?
5. **Business impact**: What would compromise of this asset enable?

At Judah Security, we built Aegis Oracle to address this gap — an exploitability scoring engine that combines CVE intelligence, runtime context, and LLM-assisted precondition analysis to produce scores that correlate with actual exploitation likelihood.

## 4. Where the Market is Getting It Wrong

### Overemphasis on CVE coverage

ASM platforms compete on the number of CVEs in their detection libraries. This is a meaningless metric. What matters is detection accuracy (low false positive rate), coverage of business logic and configuration issues, and the ability to detect novel exposure types quickly.

The most common high-impact findings in our engagements are not CVEs. They are:

- Exposed administrative interfaces (Jenkins, Kubernetes dashboards, cloud consoles)
- Development environments mistakenly exposed to the internet
- Misconfigured cloud storage with sensitive data
- Credentials or secrets in public repositories
- Overly permissive CORS configurations

None of these require a CVE assignment to be critical.

### Point-in-time thinking

Most organizations still think about security scanning as a periodic activity. The threat model has changed. Adversaries are running continuous reconnaissance. The window between asset exposure and exploitation for known vulnerability classes is often less than 48 hours after a CVE is published.

Continuous ASM is not a feature — it's the minimum viable approach.

### Integration debt

ASM tools that don't integrate with existing workflows don't get used. The best ASM deployment we've seen is one where:
- Findings flow automatically into the organization's ticketing system with enriched context
- New critical findings generate PagerDuty alerts
- Resolved tickets update the ASM platform to close findings
- Monthly reports are generated automatically for leadership

## 5. Recommendations

For organizations evaluating or maturing their ASM program:

**0–3 months:**
- Run passive subdomain enumeration against all known root domains
- Audit all cloud storage for public access
- Set up CT log monitoring for your domains
- Run Nuclei against discovered assets with critical/high severity filter

**3–6 months:**
- Establish continuous discovery pipeline
- Build asset attribution database (which host belongs to which team/application)
- Integrate findings into ticketing workflow
- Develop asset criticality scoring

**6–12 months:**
- Implement exploitability-context prioritization
- Establish SLAs for remediation by asset criticality × finding severity
- Automate evidence collection for recurring findings
- Build M&A discovery playbook

## Conclusion

ASM is not magic. The organizations that get the most value from it are the ones that treat it as an operational discipline, not a tool they deploy and forget. The discovery layer is relatively solved. The hard problems are in prioritization, workflow integration, and the organizational change management required to actually act on findings.

The vendors will keep promising AI-powered magic. The practitioners will keep doing the work.
